How JavaScript Actually Works Execution Context · Scope · Hoisting · Closures

Before writing great code, understand what the JS engine does behind the scenes. These four concepts form the backbone of every interview, every React hook, and every async bug you'll ever face.

⚙ Execution Context 🔭 Scope ⬆ Hoisting 🔒 Closures
01 — The Foundation

Execution Context

Before a single line of your code runs, JavaScript wraps it in an Execution Context — an invisible environment that holds all the variables, the value of this, and a link back to the outer world. Think of it as a self-contained room where code executes.

🧠

Phase 1 — Memory Creation

The engine scans all code first. var gets set to undefined. Function declarations get their entire body stored. let/const are registered but stay in the Temporal Dead Zone. No code runs yet.

▶️

Phase 2 — Execution

Now the engine runs your code line by line. Variables get their real values. Function calls create brand-new Execution Contexts that are pushed onto the Call Stack. When a function returns, its EC is destroyed.

🌍

Global EC

Created automatically when your script loads. Lives at the bottom of the Call Stack forever. In browsers it creates the window object and sets this = window.

⚡

Function EC

Created fresh every time a function is called. It has its own memory (variables), its own this, and a reference to the scope where it was defined — not where it was called from.

Order of Execution — What Happens & When

Step 1
Global EC Created — Memory Phase
JS engine creates the Global EC. Scans the file. Allocates undefined for var, stores function declarations fully, registers let/const in TDZ.
Step 2
Global EC Execution Phase Begins
Code runs line by line. Variables get their real values. When the engine hits a function call, it pauses and creates a new EC.
Step 3
makeCounter() Called → New EC Pushed
A fresh EC for makeCounter lands on the Call Stack. Memory phase runs inside it: count = undefined, then execution sets count = 0. The inner function increment is created — it immediately captures a reference to makeCounter's EC.
Step 4
makeCounter() Returns → EC Popped Off Stack
makeCounter's EC is removed from the Call Stack. BUT — count is NOT garbage-collected because increment still holds a reference to it. This surviving reference is the closure.
Step 5
counter() Called → increment EC Pushed
Every call to counter() creates a new EC for increment. It has no local variables, but its outer reference chain still points to makeCounter's preserved variables. count++ updates the shared closed-over value.
execution-context-makecounter.js 
let label = "global";

function makeCounter() {
  let count = 0;

  function increment() {
    count++;
    console.log(`[${label}] count = ${count}`);
  }

  return increment;
}

const c1 = makeCounter();
const c2 = makeCounter();

c1();
c1();
c2();

label = "updated";
c1();
// OUTPUT
02 — Scope

Where Variables Live

Scope is determined by the Execution Context's variable environment and the outer reference chain. The engine doesn't just look in the current EC — it walks up the chain until it finds the variable or hits global scope.

🌐

Global Scope

Declared outside any function. Lives in the Global EC's variable environment. Accessible from anywhere. Use sparingly — easy to accidentally overwrite.

📦

Function Scope

Every function call creates a fresh EC with its own variable environment. Variables declared inside cannot be seen outside. This is how you create privacy.

🔲

Block Scope

let and const are confined to their { } block. var ignores block scope — it escapes to the nearest function EC. This is a classic bug source.

GLOBAL SCOPE — Global EC's Variable Environment
let a = "global"
FUNCTION SCOPE — outer() EC
let b = "outer"sees: a ✓
FUNCTION SCOPE — inner() EC
let c = "inner"sees: a, b, c ✓
scope-chain.js 
let a = "global";

function outer() {
  let b = "outer";

  function inner() {
    let c = "inner";
    console.log(a, b, c);
  }

  inner();
}

outer();
// OUTPUT

  setTimeout(() => console.log("var:", k), 0);

  setTimeout(() => console.log("let:", m), 0);

💡

The setTimeout trap is one of the most asked questions. With var, every callback closes over the same k in the function EC — by the time setTimeout fires, the loop is done and k=3. With let, each iteration creates a fresh block-scoped variable, so each callback gets its own copy.

03 — Hoisting

The Memory Phase Effect

Hoisting is not magic. It's the visible side-effect of the EC's Memory Creation Phase. Before your code runs, the engine pre-allocates space for all declarations. What happens to that space depends on whether you used var, let/const, or a function declaration.

⚡

Why does this matter in interviews? Interviewers ask "what does this code print?" and swap the order of declarations and usages. If you understand the Memory Phase, you can trace exactly what value is in memory at each line — and never get surprised by undefined or a TypeError again.

04 — Closures

Functions That Remember

When a function is created inside another function, it gets a permanent link to the outer function's variable environment — not a copy, a live reference. This link persists even after the outer function returns and its Execution Context is gone from the stack.

What the Backpack Actually Contains

Every function object has a hidden [[scope]] property. Here's what it looks like for increment after makeCounter() returns:

📭 makeCounter() — Removed from Call Stack
Stack status→POPPED — no longer executing
count→0 (gets updated on each call)
increment→function(){...}
makeCounter() finished. Its EC is gone from the stack.
But its variable environment is kept alive in memory
because increment still holds a reference to it.
🎒 increment — The Returned Function (Carries a Backpack)
Hidden [[scope]] property automatically attached by JS engine
count→still accessible ✓
c1() called → count: 1
c1() called → count: 2
c2() called → count: 1 (own separate closure)
Each call reads & writes the SAME count in memory

private-state-pattern.js 
function createBankAccount(initialBalance) {
  let balance = initialBalance;

  return {
    deposit(amount) {
      balance += amount;
      console.log("Deposited ₹" + amount + ". Balance: ₹" + balance);
    },
    withdraw(amount) {
      if (amount > balance) {
        console.log("Insufficient funds!");
      } else {
        balance -= amount;
        console.log("Withdrew ₹" + amount + ". Balance: ₹" + balance);
      }
    },
    getBalance() {
      console.log("Balance: ₹" + balance);
    }
  };
}

const account = createBankAccount(1000);
account.deposit(500);
account.withdraw(200);
account.withdraw(5000);
account.getBalance();
console.log("Direct access:", account.balance);
// OUTPUT
memoization-with-closure.js 
function memoize(fn) {
  const cache = {};

  return function(n) {
    if (cache[n] !== undefined) {
      console.log("Cache hit for (" + n + ")!");
      return cache[n];
    }
    console.log("Computing for (" + n + ")...");
    cache[n] = fn(n);
    return cache[n];
  };
}

const square = memoize((n) => n * n);

console.log(square(5));
console.log(square(5));
console.log(square(6));
console.log(square(6));
// OUTPUT
🎒

The simplest mental model: Every function carries a backpack. The backpack contains a live reference to the variable environment of the scope where the function was written — not where it's called. When the function needs a variable, it checks its own scope first, then opens the backpack.

05 — The Big Picture

How They're All Connected

These are not four separate topics. They are four different perspectives on the same underlying machine — the Execution Context. Understand EC, and the other three click into place automatically.

⚙ Execution Context

The engine that powers all three below

🔭 Scope

Each EC has a variable environment and an outer reference. The chain of those outer references from inner to outer to global is the Scope Chain.

EC.variableEnv + outerRef = Scope Chain

⬆ Hoisting

The Memory Creation Phase of every EC is what produces hoisting. Before execution, the EC pre-allocates memory for all declarations.

EC.memoryPhase → Hoisting behavior

🔒 Closures

When a function is created inside an EC, it captures the EC's variable environment via [[scope]]. Even after the EC is popped, that reference keeps the variables alive.

fn.[[scope]] → outer EC vars = Closure
